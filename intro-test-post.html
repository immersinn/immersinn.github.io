Simpliest VSM Instantiation

Posted by TRII in text-retrieval-and-search-engines courses    Lang en

This notebook will (hopefully) be the first in a series of notebooks related to work from the Coursera course, Text Retrieval and Search Engines. For the various topics covered in the course, my goal is to impliment some of the methods and tools in order to gain some hands-on experience.

My current plan is to use the same set of data for all of the examples I construct, namly a set of articles that I have previous scraped from PhysOrg, which posts news stories related to new developments in various scientific fields. The subset of articles are from a variety of disciplnes and are pre-tagged based on categories. Evaluating query results and method performance, then, becomes a bit easier in many basic cases.

For the first article, as one can observe from the title, I will go over the process for constructing the "Simplest Vector Space Model Instantiation". Key features of this model are:

  • Bag of Word feature encoding using binary hits for terms
  • No normalization or weighting applied to terms or documents
  • Dot product similarity for calculating distances between documents / queries

Additionally, given that this is a rather small dataset, term and document preprocessing will be minimal. That is:

  • No stop word removal
  • Tokens / features are words, not phrases
  • All words to lowercase
  • Remove all punctuation
  • Remove all numbers
  • No stemming
  • No mapping multiple related words onto a single feature / word (this would likely be better implimented at the time of query anyway, as it would allow for greater flexibility)

While this limits the practical applicability of the model, it serves as a solid starting point.

In order to generate a (simpliest) Vector Space Representation for each document, we need to:

  • word tokenize the document
  • obtain the set of words contained in the document (ignore word counts)

From the overall collection, we will be able to generate a dictionary of all words. As mentioned previously, NLTK will be used for the tokenization task. While we could simply split up documents by white space, I would like to remove general punctuation and numbers from the documents. Hence, I will make use of the RegexpTokenizer. For large datasets, using regular expressions could be disastrous given the potential for regex to be quite slow (though this may be outdated). In our case, the small dataset being used allows us to make some shortcuts.

To review from above, the preprocossing should follow:

  • No stop word removal
  • Tokens / features are words, not phrases
  • All words to lowercase
  • Remove all punctuation
  • Remove all numbers
  • No stemming
  • No mapping multiple related words onto a single feature / word

The three distribution plots shown directly above allow us to compare the total number of words ina document to the number of unique words in a document. For our PhysOrg dataset, the average document has about 590 total words, approximatly 280 of which are unique.

The following is more of an aside commentary:

From the last set of data, we may be tempted to conclude that 50% of the average document is word types that occur only once. Given our current data, however, we cannot actually calculate the number of words that occur only once in the document, nor can we calculate the number of words that appear more than once in the document. In one extreme corner case for our average document, all of the 280 non-first word occurances could be attributed to a single word (i.e., one word occurs 281 times, while the remaining 279 words occur once). The other extreme case has all 280 words types in the document occurring twice such that there are no words occurring only once in the document.

Construct Dictionary and BOW Vectors¶

Steps required for this process:

  • Generate global vocabulary dictionary
  • Convert each Binary BOW Document word into its respective index value
  • Generate vector for each Document representing the words it contains

To generate the overall dictionary, we iterate over all documents and construct the complete set of words seen across all documents. We then create a look-up table (Python dictionary) to convert words to integers. This provides us with a method for transforming a word into a position along the BOW vector (i.e., the dimension in the |dictionary| - space associated with a word).

After building the dictionary, we can construct the vector corresponding to each document. In this case, we actually construct the full matrix for all documents first as a matrix of zeros. Then, iterating over each document, we replace cells corresponding to words in the documents with 1's. The result is a (very) sparse matrix of word occurances in documents. We can see that less than 0.8% of the matrix is populated with non-zeros.

Calculating Distances Between Documents¶

Summary of Doc - Doc Distance¶

In this section, we use the dot-product to compute the pair-wise distances between documents, and use the distances to find similar documents given a particular document.

Using the scipy spatial library makes computing distances a snap. The output is a list of the pair-wise distances between documents using the dot porduct as the distance metric; each pair-wise distance, then, occurs only once.

The distribution of pairwise distances looks mostly normal, with a bit of a right-ward tail, except for a (few) outliers. I would imagine the tail would grow more pronounced if we extended the documents to other domains. The min distance we encounter is 3, while the max is 256 (the x-axis has been clipped intentionally to highlight the "tail").

Inter- vs. Intra-Category Distances¶

Utilizing the pre-defined groups from PhysOrg, we can compare Doc - Doc distances for documents within the same group to distances for documents not in the same group. This will require that we performa bit more processing.

First, we need to obtain the (Document, Category) associations from the original PhysOrg article information above. Next, to facilicate indexing Doc - Doc distances, we would like to obtain a matrix for all pairwise distances -- as opposed to the "distance" vector". Following this, we will iterate over each of the Categories in the Collection to analyze and compare the intra- and inter-category document distances.

Observing the category membership counts below, we see that most documents are in the "x-news" categories. This is primarly due to the way in which the articles were "discovered" for scraping, in the sense that category labels are applied and collected differently depending on how the article was discovered by the scraper.

We will use these "Major 8" categories to compare inter / intra category distances. That is, for each category, we will use the dot product to calculate:

  • the average Document - Document distance for all documents within that category
  • the average distance from a document within the category to all documents not in the category

Upon calculating these values, we will then use a t-test to determine if the two sets of distances for each category are significantly different from one antother. While technically we should use a one-sided test if the Intra-Category distances are less than the Inter-Category distances, for this exercise the results are more interesting if we test for any difference in the means.

In [21]:
m8_lookup = [(i,c) for (i,c) in enumerate(category) if c in set(major_8)]
results01 = []
results02 = []

for i,m8 in enumerate(major_8):
    # Get documents (not) in the current category
    a = numpy.array([i for i,c in m8_lookup if c==m8])
    na = numpy.array([i for i,c in m8_lookup if c!=m8])
    
    # Get data subsets
    intra = scipy.spatial.distance.squareform(dist_mat[numpy.ix_(a, a)])
    inter = dist_mat[numpy.ix_(a, na)]
    inter = inter[numpy.triu_indices(n=inter.shape[0], m=inter.shape[1])]
    vals = numpy.concatenate((intra, inter))
    labels = ['Within' for _ in range(len(intra))] + ['Across' for _ in range(len(inter))]
    data = pandas.DataFrame({'values':vals,
                             'labels':labels})
    
    # Perform t-test
    s, p = scipy.stats.ttest_ind(intra, inter)
    
    # Update results
    results01.append({'Category':m8, 'test_stat':s, 'p_val':p,
                    'Avg_Intra':intra.mean(), 'Avg_Inter':inter.mean()})
    results02.append({'Category':m8, 'Type':'Avg_Intra', 'Value':intra.mean()})
    results02.append({'Category':m8, 'Type':'Avg_Inter', 'Value':inter.mean()})

These results are rather interesting. While all of our t-test came out significant (different intra- and inter-category mean document distances), in all cases the average distance between two documents within the same PhysOrg Category (intra) was larger than the average distance between two documents not in the same PhysOrg Category (inter)!

Considering how we pre-processed documents and how we instantiated the vectors for documents, this is perhaps an expected outcome. Recall that all words -- common and special-case words alike -- were treated exactly the same in teh model. Additionally, the only the word's presence / absence was important for creating the document vector.

In general, stop words and intermediate-commonality words are far more likley to be in any given document. Special case words, while likely quite dependent on PhysOrg category, are few and far between. Thus, a handful of common english words can completely erase the effect of common special-case words that may only appear in documents belonging only to a particular PhysOrg Category.

Conditions such as these create a situation where all documents in the collection essentially look the same, with some minor descrepencies. Using PhysOrg categories splits the collection in a seemingly arbitrary manner with regards to the document vectors. A document within any one of these arbitrary sub-groups, on average, has less in common with the sub-group than with the collection as a whole.

Wrap-Up and Next Steps¶

In this notebook, we looked at one method for implimenting the "Simplest Vector Space Model Instantiation" using Python. This simple model has three main features:

  • Bag of Word feature encoding using binary hits for terms
  • No normalization or weighting applied to terms or documents
  • Dot product similarity for calculating distances between documents / queries

A selection of PhysOrg articles was used to demonstrate the model. The results were less than positive for the Simplest VSM, as articles were, on average, more similar to articles not in their respective PhysOrg Category than to those within their category. Ideally, we would have attempted to associate documents from the non "Major 8" categories with one of the "Major 8". This particular VSM instantiation would likely not be conducive to that endeavor.

In a subsequent notebook, we will look to improve the VSM by implimenting various term weighting schemes discussed in lectures from the Text Retrieval and Search Engines Coursera course.
